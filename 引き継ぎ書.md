## 1. プロジェクトの目的と背景

日商簿記2級の合格を目指す大学生向けに、2級・3級の仕訳問題（第1問対策）をPC・タブレット・スマホで反復学習できるアプリを開発・配布するプロジェクト。依頼者（ULTRAK氏）は大学生の学習指導をしており、教材PDF（問題＋解答解説）をアプリ化して学生に配布し、大学別・学籍番号別に成績を管理者が把握できる形を目指している。

依頼者は開発者ではないが、別アプリ「コツコツサクサクアプリ」をGitHub配布＋Firebase同期コードで運用した経験がある。説明は初心者向けに丁寧に、常に日本語で行うこと。

## 4. 学生用アプリの機能仕様（v3.31時点）

単一HTMLファイル。データ保存は localStorage（キー `boki2_shiwake_20260623`）＋Firebase同期。

- **恒久的な学生識別（v3.31）**: 学籍番号をNFKC正規化し、空白を除去、英字を大文字化する。大学コードと組み合わせてSHA-256由来の決定的な `studentKey` を生成するため、同じ大学・学籍番号なら端末やブラウザに関係なく同じ `boki2_students/{studentKey}` を取得する。学生が同期コードを入力・保管する操作は廃止
- **6桁暗証番号（v3.31）**: 初回登録時に2回入力し、PBKDF2-SHA-256（ランダムsalt、120,000回）の検証値だけを保存する。別端末では学籍番号＋暗証番号で復元する。忘れた場合は警告確認後、学籍番号だけで再設定できるため、本格的な本人認証ではなく誤操作防止であることを画面に明記
- **複数端末の履歴統合（v3.31）**: 新しい解答履歴に `id` / `answeredAt`、テスト結果に `id` / `completedAt` を付ける。保存直前に端末とFirestoreの双方を統合し、Firestore RESTの更新時刻preconditionを付けて競合時は再取得・再試行する。ID付き記録は集合和、旧記録は内容指紋ごとの最大出現数で統合し、単純コピーは重複排除しながら別々の挑戦を残す
- **学籍番号修正（v3.31）**: 修正先の `studentKey` に記録があれば統合し、なければ現在の記録を新キーへ保存する。新キーへのクラウド保存成功後、旧キーには `migratedTo` / `migratedAt` と修正先情報を付けて読み取り専用にし、通常の管理一覧から除外する。旧データ自体は削除しない
- **旧端末データの救済（v3.31）**: localStorageに旧同期コードが残っている場合、その旧クラウド記録も読み込んで新学生記録へ統合する。端末データが消えている学生は、管理者による一括移行後に新方式で復元する
- **PWA（v3.31）**: Androidはインストールボタン、iPhoneは「共有→ホーム画面に追加」の案内を表示する。ナビゲーションはネットワーク優先で、通信失敗時だけキャッシュを返す。Firestore RESTやFirebase通信はサービスワーカーでキャッシュしない。PWAは起動しやすさの補助であり、本人識別は `studentKey` が担う

- **初回登録（v3.23）**: 「学籍番号」の表記を「学籍番号（大原登録番号）」に変更。任意のグループ番号を `profile.group` とFirestoreの `group` フィールドに保存する。未入力でも登録可能で、設定から後で修正できる
- **登録情報のグループ番号（v3.24）**: 設定の「現在の登録情報」枠内に「グループ番号（任意）」の入力枠と専用保存ボタンを常時表示。大学名・学籍番号の編集画面を開かなくても直接追加・変更・空欄化できる
- **大学別グループ選択（v3.25）**: 自由入力を廃止し、`boki2_sync/group-config` から所属大学の候補を読み込むプルダウンに変更。未選択は許可。既存値が候補にない場合は「現在の登録」として保持する
- **解答画面の操作ボタン（v3.26）**: 「採点する」を「解答確認」に改称し、白背景・薄い枠線の控えめな表示に変更。「前へ」「次の問題へ」「全体の結果へ」はメインカラーで塗りつぶし、問題移動の導線を強調する
- **未入力時の解答確認（v3.27）**: 借方・貸方が未入力、または片側だけの入力でも警告で止めず、未回答（不正解）として正解仕訳と解説を表示する
- **管理者による利用停止（v3.31で新方式へ更新）**: `boki2_access/{studentKey}` の `suspended` がtrueの場合、閉じられない案内画面を表示し、担当教員への確認を促す。学生側は名簿全体を取得しない
- **3級直前模試の追加（v3.29）**: 2025年度3級直前模試第1回〜第4回の第1問1〜15、計60問を追加。各問3点、各回45点。既存2級問題と同じ同期・公開設定基盤で扱い、ホームの回別ボタンは「2級模試／3級模試・回・第1問等」を識別できる表示にした。古いクラウド記録を復元・取得した後にも同梱セットを再統合し、新規セットが消えないようにしている
- **ホーム（v3.31）**: ①回答テスト①〜⑨のメインボタン（直近点数表示、70%以上で緑）②その他の練習③⚙ 設定。設定内では現在の大学・学籍番号・グループ、クラウド同期状態、暗証番号変更、引き継ぎガイド、PWA案内を表示する。学籍番号修正時は修正先の学生記録と履歴を統合する
  - **管理者確認モード（v3.18）**: URLに `?preview=admin` を付けると、初回登録ポップアップを出さずに学生画面を確認できる。確認モードでは端末内に登録情報が残っていてもクラウド保存を止めるため、確認用の解答は管理者ダッシュボード集計へ上がらない
  - **大学別URL / QRコード（v3.16）**: 学生用URLに `?univ=gakushuin` のような大学コードを付けると、その大学専用リンクとして扱う。初回登録ポップアップでは大学名を選ばせず「所属大学：学習院大学」と固定表示し、学生は学籍番号だけ入力する。設定内の修正画面でも大学名はそのリンクの大学に固定される。通常URL（`?univ=`なし）では従来どおり大学選択＋その他入力が使える。対応表は `UNIVERSITY_LINKS`（学生アプリ）と管理画面の同名定数で管理。現時点のコード例：`gakushuin` / `nichidai-econ` / `nichidai-commerce` / `toyo` / `komazawa` / `senshu`
  - **初回起動時の必須登録（v3.7、v3.14で修正導線を明確化、v3.16で大学固定リンク対応）**: 未登録なら `requireProfile()` が閉じられないポップアップで大学・学籍番号の入力を促す。大学別URLの場合は大学名を固定し、学籍番号だけを入力させる（`commitProfile()` で保存）。以後は「設定 → データ保存・引き継ぎ → ユーザー情報を修正する」から変更可
  - **旧同期コード方式（v3.8〜v3.29・履歴）**: 以前は登録時に同期コードを自動発行し、機種変更時に学生が再入力していた。v3.31で学生操作から廃止し、旧ドキュメントは移行元としてのみ使用する
- **問選択ポップアップ**: テスト①〜⑨を押すと問1〜5を複数選択して出題（未選択の問は「未解答」のまま。配点も選択分の合計満点 ＝ ダッシュボードで分母が人により変わる理由）
- **管理者からの課題（自動配信・v3.10）**: 管理者がダッシュボードで作った課題（①〜⑨から選んだ問のセット）が、対象（全員／大学／学籍番号）に合致する生徒のホーム上部に「📌 管理者からの課題」カードとして**自動で表示**（コード入力不要）。押すと採点付きで出題し、結果は同期でダッシュボードに集約。実装：起動時とプロフィール保存時に `loadAssignments()`（`boki2_sync/assignments-config` をRESTでGET・**読むだけ**）→ `myAssignments()`（`store.profile` で対象判定）→ `renderAssignments()`。出題は `startAssignment(id)`（`startPicked` を雛形に qid配列から採点セッション、`groupName="課題:"+title`, `assignmentId` を保持）。結果は `finish()` が `store.tests` に `assignmentId` と問別⭕❌の `detail[]` を付けて記録
- **Study Journal UI（v3.22）**: 学生のスマートフォン縦画面を基本に、日付入りヘッダー、紙面風の見出し、下部固定ナビゲーション、2列のテスト索引、編集的な成績画面へ刷新。問題選択・採点・同期・公開制御のロジックは変更していない
- **問題公開設定（v3.21〜）**: `boki2_sync/publications-config` をRESTで読み、対象（全員／大学／大学＋学籍番号）と公開開始・終了日時が一致する問題だけを表示する。2025年度2級模試48問と3級模試60問（計108問）は `CONTROLLED_QIDS` で初期状態から管理対象とし、設定が未取得・未作成でも非表示（安全側）。従来49問は互換性のため従来どおり表示する。設定はlocalStorageへキャッシュし、開始・終了時刻に画面を自動更新する
- **解答画面**: 記号は10文字幅プルダウン、金額はカンマ込み8桁幅＋datalist候補＋直接入力。都度採点もまとめ採点も可。採点後の解説は文字を必須とし、`explanationImages` がある場合はその下に確認済みPNGとキャプションを表示（v3.19）。画像パスは `assets/explanations/` 配下のPNGだけを許可し、未確認画像は表示しない。全問後に選択した問題の合計点（2級は原則4点、3級模試は3点）と⭕❌明細。**画面下部に「■ 中断してホームに戻る」ボタン（v3.6）**：押すと3択ポップアップ（保存して中断／保存せずにやめる／キャンセル）。「保存して中断」で解答中セッションを保持しホームの「▶ 解答中の問題に戻る」から再開可（※実装は `confirmQuit()`／`session` はメモリ保持のみ。**アプリを閉じる・再読み込みすると途中の答えは消える**仕様。「保存せずにやめる」は `session=null`）
- **解答ストップウォッチ（v3.9）**: 各問題の所要時間を計測。問題を開くと画面上部（進捗表示の隣）に `⏱ mm:ss` のストップウォッチが表示され経過時間をカウント。**採点した時点でその問題の時間を確定（凍結）**、次の問題へ進むと次問はゼロから再計測。全問終了後のサマリーに **トータル所要時間** と各問題の所要時間を表示。**別タブ・バックグラウンドに離れている間はタイマー一時停止**（`visibilitychange` で `pauseTimer`／復帰で `beginTimer`）。中断→キャンセルや「解答中の問題に戻る」でも再開する。実装は `beginTimer`／`pauseTimer`／`swDisplay`／`fmtClock`／`fmtDur`／`fmtDiff`（`session.times[]`/`tFrozen[]`/`tStart`/`tIdx` に計測状態を保持。`session` はメモリ保持のみのため再読み込みで消える点は従来同様）
- **成績**: サマリー → 回別テスト成績（棒グラフ・前回比／**総所要時間と前回比（±分秒・速いほど緑＝改善）も表示 v3.9**）→ 案Bヒートマップ → 案C分野別成長比較 → 日別チャート → 分野別/問題別（**問題別に平均所要時間を併記 v3.9**）→ 成績レポート保存＋印刷/PDF
- **問題データ形式（JSON）**: `{name, questions:[{questionId?, label(必須), category, topic, text, options, debit[[記号,金額]..], credit[..], expl, explanationImages?, group, points, sourcePages?}]}`。2級模試の `topic` は `skills/extract-boki-mock-pdf/references/title-map.json` の固定タイトルを使用する。`explanationImages` は確認済みPNGの `src`、`alt`、`caption`、出典ページ、切り抜き座標、寸法、SHA-256等を持つ配列（画像不要なら `[]`）。取込時に貸借一致・記号整合を自動検証。IDは `questionId` を優先し、無い場合だけ `セット名_qN` で自動採番するため、模試には固定ASCII IDを必ず付ける
- **学生ドキュメント構造**（`boki2_students/{studentKey}`）:
  - `univ` / `univCode` / `sid` / `sidNormalized` / `group` / `pinSalt` / `pinHash` / `pinIterations` / `schemaVersion` / `createdAt` / `updatedAt` / `data`
  - `data` の中身 = `{history, sets, meta, tests, profile, updatedAt}`
  - `history` = `[{id,qid,correct,date,answeredAt,ms?}]`。旧記録では `id` / `answeredAt` がなくても統合可能
  - `tests` = `[{id,group,score,max,correct,total,date,completedAt,timeMs?,assignmentId?,detail?}]`。旧記録では `id` / `completedAt` がなくても統合可能
  - `boki2_access/{studentKey}` = `{active,suspended,univ,univCode,sid,sidNormalized,group,updatedAt}`。学生は自分の1件だけ読み、管理者だけが一覧・更新する
  - `boki2_sync/{旧同期コード}` は移行元。移行後は `migratedTo` / `migratedAt` を付けて読み取り専用で保管する
  - **課題定義ドキュメント**（`boki2_sync/assignments-config`）: 管理者が書き（生徒は読むだけ）。中身 `data`（JSON文字列）= `{assignments:[{id,title,target,qids,createdAt}], updatedAt}`。`target`＝`{mode:"all"}`／`{mode:"univ",univ}`／`{mode:"sids",sids:[]}`。大学別管理者は画面上、担当大学向けの課題のみ作成・編集・結果確認できる。`qids` は `第1問特集①〜⑨_qN` 形式。`assignments-config` の `create/update` はFirestoreルールで管理者限定（生徒アプリからの `get` は従来どおり全員許可）。ダッシュボードは `list` 取得時にこの `doc.id`（= `ASSIGN_DOC`）を学生行から除外する。⚠️**ドキュメントIDに `__..__`（前後アンダースコア2つ）は使えない**（Firestoreの予約ID。v1.3までは `__assignments__` で保存エラーになっていたため v1.4 で `assignments-config` に変更）
  - **問題公開設定ドキュメント**（`boki2_sync/publications-config`、v3.21〜／管理画面v1.18）: `data`（JSON文字列）= `{publications:[{id,title,ownerEmail,target,qids,enabled,publishFrom,publishUntil,createdAt,updatedAt}],updatedAt}`。大学別管理者は自分が所有し、担当大学を対象とする設定だけを編集できる。保存・削除はFirestore transactionで競合を抑える。学生は読むだけ
  - **管理者権限設定ドキュメント**（`boki2_sync/admin-config`、v1.7）: マスター管理者だけが管理画面の「🔐 管理者権限設定」から作成・更新できる。中身 `data`（JSON文字列）= `{admins:[{email,role,univs,createdAt}], updatedAt}`、Firestoreルール判定用に `adminsMap` も同時保存。`role` は `master` または `univ`、大学別管理者の `univs` は担当大学名配列。`admin-config` は学生行・課題行から除外する

## 7. 開発時の作法（次のエージェントへ）

- 編集はマスター版 `src/仕訳学習アプリ.html` に行い、ヘッダーのバージョン表記（v3.x）を必ず上げる
- 配布版の再生成: リポジトリ直下で `node scripts/build-distribution.mjs` を実行する。問題JSON注入、配布版表記、相対パスの調整をスクリプトが行う
- 動作検証は jsdom で実施してきた。scrollIntoView / requestAnimationFrame / clipboard はスタブが必要。`let session` 等はevalスコープ内のため、テストコードはアプリJSと同一eval内で実行すること
- エラーは画面上部の赤いバナーに表示される（window.onerror）。依頼者にバナー文言を聞けば調査可能

## 8. 既知の注意点・リスク

- v3.31では同じ大学・学籍番号から同じクラウド記録を取得するため、ブラウザ内保存が消えても学籍番号と暗証番号で復元できる。ただし、初回保存が完了する前に閉じた端末内だけの記録は復元できない
- 旧版の記録は、旧localStorageが残る学生なら新アプリ初回接続時に統合される。端末側も消えている学生は、管理画面の一括移行が終わるまで新方式から旧履歴を復元できない
- **解答中の「中断→保存して中断」で保持されるのはメモリ上のセッションのみ（v3.6）**。同じ画面を開いている間は「解答中の問題に戻る」で再開できるが、タブを閉じる・再読み込みすると途中の（未採点の）答えは失われる。将来「後日も続きから」に対応したい場合は `session` を localStorage に一時保存する実装追加が必要（依頼者の選択で今回はメモリ保持のみに決定）
- **1問ごとに「採点する」を押した時点で正誤が履歴に即保存される**。そのため「保存せずにやめる（破棄）」を選んでも、すでに採点済みの問題は分野別・問題別・復習の集計に残る（回別テストの合計点は「全体の結果へ」到達時のみ記録なので、中断すれば中途半端な点数は残らない）
- 暗証番号は学籍番号だけで再設定でき、強い本人認証ではない。他人の学籍番号を知る学生が再設定して記録へ入る可能性がある。教師対応不要を優先した明示的な設計判断
- `studentKey` は大学コードと学籍番号から決定的に算出できるため、学籍番号を知る第三者に対する秘密情報ではない。学生一覧取得はルールで禁止するが、成績改ざん耐性が必要なら将来Firebase Authentication等の本人認証が必要
- 教材由来の問題文・必要図解について、**教材権利者から本アプリでの利用許可を取得済み**（2026-07-10、依頼者が再確認）。PDF原本や全ページ画像は公開リポジトリへ置かず、アプリ用に確認済みの問題データと必要箇所だけの切り抜き画像を扱う
- **問題公開設定は画面上の表示制御**。問題本文が配布版HTMLや端末内データに含まれる構成では、開発者ツール等による完全な秘匿・公開後のオフライン端末からの回収はできない。厳密な機密配信が必要なら、問題本文自体を認証付きAPIから公開時に取得する構成へ変更する
- 成績はブラウザ→Firestoreの自己申告ベース（改ざん耐性なし）。割り切り
- GitHub Pages上のアプリとローカルで開いたHTMLはlocalStorageが別だが、同じ大学・学籍番号と暗証番号で同じクラウド記録を取得できる
- ダッシュボードの①〜⑨列の分母は学生ごとに変わりうる（問選択ポップアップで受験した問だけが満点になるため。仕様どおり）
- 管理者ページは公開URL上にあるが、データ取得はログイン＋ルールで保護されるため他人には中身が見えない
